# HU-02: Registro de Tienda Verificada

## Objetivo

- Actor: propietario de tienda de hardware
- Capacidad: registrar el negocio con datos comerciales y documento de soporte
- Resultado esperado: iniciar el proceso de verificacion administrativa de la
  tienda

## Interfaz

| Metodo | Ruta | Finalidad |
|--------|------|-----------|
| `POST` | `/api/v1/auth/register/tienda` | Registrar tienda |

El endpoint usa `multipart/form-data`.

## Entrada

- `nombre_comercial`
- `ruc`
- `direccion`
- `telefono`
- `correo`
- `password`
- `documento_identidad`

## Flujo principal

1. validar campos de texto requeridos
2. validar correo, telefono, RUC y contrasena
3. validar archivo adjunto y mimetype real
4. verificar unicidad de correo, telefono, RUC y nombre comercial
5. generar hash de contrasena
6. renombrar el documento con nombre interno seguro
7. crear usuario con `rol = TIENDA_VERIFICADA`
8. asignar `estado = EN_REVISION`
9. crear registro relacionado en `tiendas`
10. persistir archivo solo si la transaccion de datos fue exitosa
11. disparar el flujo de verificacion de correo

## Datos relevantes

### Tabla implicada

- `tiendas`

## Respuesta esperada

`201 Created` con identificador del usuario, datos comerciales base, `rol` y
`estado`.

## Errores

| Codigo | Situacion |
|--------|-----------|
| `400` | campos o documento ausentes |
| `409` | correo, telefono, RUC o nombre comercial duplicado |
| `413` | documento supera el tamano permitido |
| `415` | formato de archivo no permitido |
| `422` | correo, telefono, RUC o contrasena invalida |

## Reglas

- la verificacion de correo no activa automaticamente la tienda
- la activacion final depende del flujo administrativo de HU-20
- la ruta del documento se guarda como relativa
- el nombre original del archivo no debe persistirse
