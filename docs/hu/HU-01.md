# HU-01: Registro de Usuario Estándar

**Como** ciudadano de Ayacucho
**Quiero** registrarme con correo y contraseña
**Para** publicar y explorar anuncios de hardware.

## Endpoint

POST `/api/v1/auth/register`

## Entrada

- nombre (max 100 chars)
- correo (formato email válido)
- password
- telefono (9 dígitos, solo numéricos)

## Flujo (orden obligatorio)

1. Validar campos con Marshmallow (sino 400).
2. Validar formato correo (sino 422).
3. Validar complejidad password: mínimo 8 chars, al menos 1 mayúscula, 1 número, 1 carácter especial (sino 422).
4. Validar teléfono: exactamente 9 dígitos numéricos (sino 422).
5. Verificar correo único en BD (sino 409).
6. Verificar teléfono único en BD (sino 409).
7. Cifrar password con bcrypt (salt=10).
8. INSERT usuario con `rol=USER_ESTANDAR`, `estado=PENDIENTE_VERIFICACION`.
9. Disparar flujo de verificación de correo (HU-03) de forma async.
10. Si falla INSERT → rollback.

## Respuesta

**201 Created**
```json
{
  "id": 1,
  "nombre": "Juan Quispe",
  "correo": "juan@gmail.com",
  "rol": "USER_ESTANDAR",
  "estado": "PENDIENTE_VERIFICACION",
  "mensaje": "Revisa tu correo para activar tu cuenta."
}
```

## Errores

- 400 Campos obligatorios ausentes.
- 409 Correo ya registrado / teléfono ya registrado.
- 422 Formato de correo inválido / teléfono no es 9 dígitos / password no cumple requisitos de complejidad.

## Reglas

- Correo único — constraint en BD + verificación previa en Marshmallow.
- Teléfono único — constraint en BD + verificación previa en Marshmallow, exactamente 9 dígitos numéricos.
- Password nunca se expone en ninguna respuesta.
- Complejidad de password: mínimo 8 chars, al menos 1 mayúscula, 1 número, 1 carácter especial — consistente con HU-02.
- `estado=PENDIENTE_VERIFICACION` hasta completar HU-03.
- Cuenta en `PENDIENTE_VERIFICACION` no puede hacer login ni publicar anuncios.
- Envío de correo de verificación async — fallo de envío no revierte el registro, solo loggea.
- Nunca exponer password en ninguna respuesta.
- `rol` y `estado` se exponen en este endpoint porque es el propio usuario viendo su información — no aplica la regla de transformación de endpoints públicos.
