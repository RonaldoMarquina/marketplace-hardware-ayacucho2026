# HU-07: Edicion de Anuncio Propio

## Objetivo

- Actor: vendedor autenticado
- Capacidad: editar, desactivar y reactivar anuncios propios, asi como gestionar
  la media asociada
- Resultado esperado: mantener actualizada la informacion del anuncio sin perder
  control sobre su disponibilidad

## Interfaz

| Metodo | Ruta | Finalidad |
|--------|------|-----------|
| `PATCH` | `/api/v1/anuncios/:anuncio_id` | Editar campos del anuncio |
| `PATCH` | `/api/v1/anuncios/:anuncio_id/desactivar` | Desactivar anuncio |
| `PATCH` | `/api/v1/anuncios/:anuncio_id/reactivar` | Reactivar anuncio |
| `PATCH` | `/api/v1/anuncios/:anuncio_id/media/orden` | Reordenar imagenes |
| `DELETE` | `/api/v1/anuncios/:anuncio_id/media/:media_id` | Eliminar imagen o video |
| `PUT` | `/api/v1/anuncios/:anuncio_id/media/:media_id` | Reemplazar imagen o video |

Todas las operaciones requieren JWT valido y propiedad sobre el anuncio.

## Operacion 1: edicion de campos

### Campos editables

- `titulo`
- `categoria`
- `subcategoria`
- `descripcion`
- `condicion`
- `precio`
- `especificaciones`

### Campos no editables

- `usuario_id`
- `estado`
- `created_at`

Si estos campos llegan en el body, deben ignorarse sin modificar el recurso.

### Flujo principal

1. validar JWT
2. verificar existencia del anuncio
3. verificar propiedad del anuncio
4. permitir la operacion solo si `estado = ACTIVO`
5. rechazar body vacio
6. validar consistencia entre `categoria` y `subcategoria`
7. validar el resto de campos con schema
8. aplicar merge parcial sobre `especificaciones` cuando corresponda
9. actualizar solo campos presentes en la solicitud
10. registrar auditoria funcional de la modificacion

### Respuesta esperada

`200 OK` con el anuncio actualizado y `estado` actual.

## Operacion 2: desactivacion

### Flujo principal

1. validar JWT
2. verificar existencia
3. verificar propiedad
4. permitir la operacion solo si `estado = ACTIVO`
5. actualizar `estado = INACTIVO`
6. actualizar `updated_at`

### Respuesta esperada

`200 OK` con `id`, `estado` y `updated_at`.

## Operacion 3: reactivacion

### Flujo principal

1. validar JWT
2. verificar existencia
3. verificar propiedad
4. permitir la operacion solo si `estado = INACTIVO`
5. verificar `reactivaciones_count < 3`
6. actualizar `estado = ACTIVO`
7. incrementar `reactivaciones_count`
8. actualizar `updated_at`

### Respuesta esperada

`200 OK` con `id`, `estado`, `reactivaciones_count`,
`reactivaciones_restantes` y `updated_at`.

## Operacion 4: reordenamiento de imagenes

### Entrada

```json
{ "orden": [3, 1, 2] }
```

### Flujo principal

1. validar JWT y propiedad
2. permitir la operacion solo si `estado = ACTIVO`
3. verificar que todos los `media_id` pertenezcan al anuncio
4. excluir videos del arreglo de orden
5. exigir que el arreglo contenga exactamente todas las imagenes sin repetir
6. reasignar `orden`
7. marcar como principal la primera imagen del arreglo

### Respuesta esperada

`200 OK` con la lista de media en el nuevo orden.

## Operacion 5: eliminacion de media

### Flujo principal

1. validar JWT y propiedad
2. permitir la operacion solo si `estado = ACTIVO`
3. verificar que `media_id` pertenezca al anuncio
4. si la imagen eliminada es principal, promover una nueva imagen principal si
   existe
5. eliminar registro en base de datos
6. eliminar archivo fisico
7. reordenar las imagenes restantes

### Respuesta esperada

`200 OK` con la media eliminada y la media restante.

## Operacion 6: reemplazo de media

### Restricciones de archivo

- imagenes: `JPG`, `JPEG`, `PNG`, `WEBP`, `AVIF`, maximo `10MB`
- videos: `MP4`, maximo `40MB`
- el archivo nuevo debe conservar el mismo `tipo_media`

### Flujo principal

1. validar JWT y propiedad
2. permitir la operacion solo si `estado = ACTIVO`
3. verificar pertenencia de `media_id`
4. validar mimetype real del archivo nuevo
5. guardar nuevo archivo con nombre interno controlado
6. actualizar `ruta_relativa`
7. eliminar el archivo anterior

### Respuesta esperada

`200 OK` con el registro de media actualizado.

## Operacion delegada

La incorporacion de nueva imagen o video no se resuelve en esta HU.

- referencia: HU-06
- endpoint: `POST /api/v1/anuncios/:anuncio_id/media`

## Errores

| Codigo | Situacion |
|--------|-----------|
| `400` | body vacio o datos incompletos |
| `403` | usuario sin propiedad o recurso bloqueado |
| `404` | anuncio o media inexistente |
| `409` | estado del anuncio no compatible con la operacion |
| `413` | archivo supera el limite permitido |
| `422` | validacion de categoria, subcategoria, precio, JSON o tipo de archivo |

## Reglas

- todas las operaciones verifican propiedad antes de modificar datos o archivos
- el vendedor no puede modificar anuncios `BLOQUEADO` ni `VENDIDO`
- las operaciones sobre media requieren `estado = ACTIVO`
- `created_at` no se modifica durante reactivaciones
- al agotar reactivaciones, el anuncio no debe volver a `ACTIVO`
- la consistencia entre base de datos y archivos debe preservarse como
  prioridad operativa
