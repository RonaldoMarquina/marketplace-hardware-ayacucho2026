# HU-07: Edición de Anuncio Propio

**Como** vendedor autenticado
**Quiero** editar, desactivar y reactivar mis anuncios publicados
**Para** corregir errores, actualizar información o gestionar su disponibilidad.

---

## 1. Edición de campos del anuncio

PATCH `/api/v1/anuncios/:anuncio_id`
🔒 JWT requerido

### Campos editables

titulo | categoria | subcategoria | descripcion | condicion | precio | especificaciones

### Campos inmutables

usuario_id | estado | created_at
→ Ignorar si llegan en el body sin retornar error.

### Flujo (orden obligatorio)

1. Validar JWT.
2. Verificar anuncio existe (sino 404).
3. Verificar propietario: `usuario_id` del JWT == `anuncio.usuario_id` (sino 403).
4. Verificar estado == ACTIVO (VENDIDO → 409, INACTIVO → 409, BLOQUEADO → 403).
5. Verificar body no vacío (sino 400).
6. Si viene `categoria` sin `subcategoria` → 422.
7. Si viene `categoria` + `subcategoria` → validar que `subcategoria` pertenece a la nueva `categoria` (sino 422).
8. Si viene solo `subcategoria` → validar contra la `categoria` actual del anuncio (sino 422).
9. Si viene `especificaciones` → aplicar `JSON_MERGE_PATCH` sobre el JSON existente.
   - Campos con valor `null` explícito se eliminan del JSON.
   - Campos ausentes se conservan intactos.
10. Validar resto de campos con Marshmallow.
11. UPDATE solo campos presentes en body + `updated_at=NOW()`.
12. Log: usuario_id, anuncio_id, campos modificados, timestamp.

### Respuesta

**200 OK**
```json
{
  "id": 15,
  "titulo": "Ryzen 5 5600X",
  "categoria": "COMPONENTES",
  "subcategoria": "Procesador",
  "condicion": "COMO_NUEVO",
  "precio": 380.00,
  "descripcion": "Precio rebajado, negociable",
  "especificaciones": {
    "socket": "AM4",
    "nucleos": 6,
    "velocidad_base_ghz": 3.7,
    "incluye_cooler": true
  },
  "estado": "ACTIVO",
  "updated_at": "2026-01-01T00:00:00"
}
```

### Errores

- 400 Body vacío.
- 403 No es propietario / anuncio BLOQUEADO.
- 404 Anuncio no existe.
- 409 Anuncio VENDIDO o INACTIVO.
- 422 `categoria` sin `subcategoria` / `subcategoria` no válida para la categoría / precio inválido / precio con más de 2 decimales / título > 100 chars / condicion fuera de ENUM / `especificaciones` no es JSON válido.

### Reglas

- PATCH: solo actualizar campos presentes en body.
- `usuario_id` siempre del JWT, nunca del body.
- `categoria` es editable pero exige `subcategoria` en el mismo body.
- `especificaciones` usa `JSON_MERGE_PATCH`: merge parcial, `null` elimina campo.
- `condicion` debe pertenecer al ENUM si se envía.
- Precio DECIMAL(10,2), rechazar más de 2 decimales.

## 2. Desactivar anuncio (borrado lógico)

PATCH `/api/v1/anuncios/:anuncio_id/desactivar`
🔒 JWT requerido

### Flujo

1. Validar JWT.
2. Verificar anuncio existe (sino 404).
3. Verificar propietario (sino 403).
4. Verificar estado == ACTIVO (BLOQUEADO → 403, VENDIDO → 409, ya INACTIVO → 409).
5. UPDATE `estado = INACTIVO` + `updated_at = NOW()`.
6. Log: usuario_id, anuncio_id, acción=DESACTIVAR, timestamp.

### Respuesta

**200 OK**
```json
{
  "id": 15,
  "estado": "INACTIVO",
  "updated_at": "2026-01-01T00:00:00"
}
```

### Errores

- 403 No es propietario / anuncio BLOQUEADO.
- 404 Anuncio no existe.
- 409 Anuncio ya INACTIVO / anuncio VENDIDO.

### Reglas

- Archivos físicos (imágenes/video) se conservan en disco.
- El anuncio desaparece del feed inmediatamente.
- El historial y media se preservan para posible reactivación.
- Anuncio BLOQUEADO por admin no puede ser desactivado desde este endpoint.

## 3. Reactivar anuncio

PATCH `/api/v1/anuncios/:anuncio_id/reactivar`
🔒 JWT requerido

### Flujo

1. Validar JWT.
2. Verificar anuncio existe (sino 404).
3. Verificar propietario (sino 403).
4. Verificar estado == INACTIVO (BLOQUEADO → 403, ACTIVO → 409, VENDIDO → 409).
5. Verificar `reactivaciones_count` < 3 (sino 403).
6. UPDATE `estado = ACTIVO` + `reactivaciones_count += 1` + `updated_at = NOW()`.
7. Log: usuario_id, anuncio_id, acción=REACTIVAR, reactivaciones_count, timestamp.

### Respuesta

**200 OK**
```json
{
  "id": 15,
  "estado": "ACTIVO",
  "reactivaciones_count": 1,
  "reactivaciones_restantes": 2,
  "updated_at": "2026-01-01T00:00:00"
}
```

### Errores

- 403 No es propietario / anuncio BLOQUEADO / límite de 3 reactivaciones alcanzado.
- 404 Anuncio no existe.
- 409 Anuncio ya ACTIVO / anuncio VENDIDO.

### Reglas

- `created_at` se conserva intacto — el anuncio no gana posición en el feed por reactivarse.
- Máximo 3 reactivaciones por anuncio de por vida.
- Al agotar las 3 reactivaciones el anuncio queda `INACTIVO` permanente.
- El usuario debe republicar desde HU-05 si quiere volver a listar el producto.
- Anuncio BLOQUEADO por admin no puede reactivarse desde este endpoint.

## 4. Reordenar imágenes

PATCH `/api/v1/anuncios/:anuncio_id/media/orden`
🔒 JWT requerido

### Body

```json
{ "orden": [3, 1, 2] }
```
→ Array de `media_id` en el nuevo orden deseado. El primero pasa a `es_principal=True`.

### Flujo

1. Validar JWT y propietario (sino 403).
2. Verificar estado == ACTIVO (sino 409/403).
3. Verificar que todos los IDs pertenecen al anuncio (sino 422).
4. Verificar que el array no incluye el ID del video (sino 422).
5. Verificar que el array contiene todos los IDs de imágenes del anuncio, sin omitir ni repetir (sino 422).
6. UPDATE `orden` y `es_principal` en `media_anuncio`.

### Respuesta

**200 OK**
```json
{
  "media": [
    { "id": 3, "tipo_media": "imagen", "ruta_relativa": "/uploads/anuncios/15/uuid-ts.jpg", "es_principal": true, "orden": 0 },
    { "id": 1, "tipo_media": "imagen", "ruta_relativa": "/uploads/anuncios/15/uuid-ts.jpg", "es_principal": false, "orden": 1 },
    { "id": 2, "tipo_media": "imagen", "ruta_relativa": "/uploads/anuncios/15/uuid-ts.jpg", "es_principal": false, "orden": 2 }
  ]
}
```

### Errores

- 403 No es propietario.
- 409 Anuncio no ACTIVO.
- 422 Array contiene IDs ajenos / incluye ID de video / faltan IDs / IDs repetidos.

## 5. Eliminar imagen o video

DELETE `/api/v1/anuncios/:anuncio_id/media/:media_id`
🔒 JWT requerido

### Flujo

1. Validar JWT y propietario (sino 403).
2. Verificar estado == ACTIVO (sino 409/403).
3. Verificar que `media_id` pertenece al anuncio (sino 404).
4. Si es imagen y `es_principal=True` → promover imagen con `orden=1` a principal antes de eliminar.
5. Si es la única imagen → eliminar sin reasignar principal (anuncio queda sin imagen).
6. DELETE en `media_anuncio`.
7. Eliminar archivo físico del disco.
8. Si falla eliminación física → revertir DELETE en BD (rollback manual).
9. Reordenar `orden` de imágenes restantes para mantener secuencia sin huecos.

### Respuesta

**200 OK**
```json
{
  "eliminado": { "id": 2, "tipo_media": "imagen" },
  "media_restante": [
    { "id": 1, "tipo_media": "imagen", "ruta_relativa": "/uploads/anuncios/15/uuid-ts.jpg", "es_principal": true, "orden": 0 }
  ]
}
```

### Errores

- 403 No es propietario.
- 404 `media_id` no pertenece al anuncio.
- 409 Anuncio no ACTIVO.

## 6. Reemplazar imagen o video existente

PUT `/api/v1/anuncios/:anuncio_id/media/:media_id`
🔒 JWT requerido | Content-Type: `multipart/form-data`

### Restricciones

- El archivo nuevo debe ser del mismo `tipo_media` que el que reemplaza.
- Imágenes: JPG, PNG, WEBP — máx 10MB.
- Videos: MP4 — máx 40MB.

### Flujo

1. Validar JWT y propietario (sino 403).
2. Verificar estado == ACTIVO (sino 409/403).
3. Verificar `media_id` pertenece al anuncio (sino 404).
4. Validar mimetype real del archivo nuevo.
5. Guardar archivo nuevo en disco con nuevo `{uuid}-{timestamp}.{ext}`.
6. UPDATE `ruta_relativa` en `media_anuncio`.
7. Eliminar archivo físico anterior.
8. Si falla paso 7 → loguear para limpieza diferida (no revertir, BD ya actualizada).

### Respuesta

**200 OK**
```json
{
  "id": 2,
  "tipo_media": "imagen",
  "ruta_relativa": "/uploads/anuncios/15/nuevo-uuid-ts.jpg",
  "es_principal": false,
  "orden": 1
}
```

### Errores

- 403 No es propietario.
- 404 `media_id` no pertenece al anuncio.
- 409 Anuncio no ACTIVO.
- 413 Archivo supera límite de tamaño.
- 422 Tipo de archivo inválido / tipo_media no coincide con el original.

## 7. Agregar nueva imagen o video

→ Usar el endpoint de **HU-06** directamente.
`POST /api/v1/anuncios/:anuncio_id/media`

---

## Reglas generales

- Todas las operaciones verifican propietario antes de cualquier acción sobre BD o disco.
- Anuncio BLOQUEADO por admin: ninguna operación de esta HU puede modificarlo.
- Anuncio VENDIDO: ninguna operación de esta HU puede modificarlo.
- Archivos físicos y registros en BD deben mantenerse siempre sincronizados; ante duda, priorizar consistencia en BD y loguear archivos huérfanos para limpieza diferida.
- Las operaciones sobre media (secciones 4–7) requieren estado ACTIVO.
- `/uploads/` en `.gitignore`.
- Campo `reactivaciones_count` vive en la tabla `anuncios`, valor inicial 0 al momento del INSERT (HU-05).
