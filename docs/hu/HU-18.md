# HU-18: Historial de Transacciones del Usuario

**Como** usuario autenticado
**Quiero** ver mi historial de compras y ventas
**Para** hacer seguimiento de mis transacciones y calificaciones pendientes.

## Endpoint

GET `/api/v1/usuarios/me/transacciones`
🔒 JWT requerido

## Query params

| Param | Valores | Default |
|-------|---------|---------|
| tipo  | `ventas` \| `compras` \| `ambas` | `ambas` |
| page  | Entero > 0 | 1 |
| limit | Entero 1–20 | 10 |

## Flujo

1. Verificar JWT válido + cuenta `ACTIVO` (sino 401/403).
2. Validar query params (sino 400/422).
3. Según `tipo`:
   - `ventas` → `WHERE vendedor_id = :usuario_id`.
   - `compras` → `WHERE comprador_id = :usuario_id`.
   - `ambas` → `WHERE vendedor_id = :usuario_id OR comprador_id = :usuario_id`.
4. LEFT JOIN `anuncios` → obtener `titulo`, `categoria`, `subcategoria`, `imagen_principal`.
5. LEFT JOIN `usuarios` (vendedor y comprador) → obtener nombres.
6. LEFT JOIN `calificaciones` → determinar `calificacion_pendiente` para cada transacción.
7. Ordenar `ORDER BY created_at DESC`.
8. Paginar con OFFSET.

## Errores

- 401 Sin JWT o token inválido.
- 403 Cuenta no `ACTIVO`.
- 422 `tipo` fuera de valores permitidos.

## Reglas

- Endpoint privado — solo el propio usuario ve su historial, nunca el de otro.
- `usuario_id` siempre del JWT, nunca del path ni del body.
- `tipo` en cada item es `"venta"` o `"compra"` — indica el rol del usuario autenticado en esa transacción.
- Un mismo usuario puede aparecer como vendedor en unas y comprador en otras — `ambas` las mezcla ordenadas por fecha.
- `contraparte` es el otro participante de la transacción:
  - En ventas → el comprador.
  - En compras → el vendedor.
- `calificacion_pendiente: true` → el usuario autenticado aún no calificó a la contraparte en esa transacción.
- `calificacion_emitida` es la calificación que el usuario autenticado emitió — `null` si aún no calificó.
- `calificacion_pendiente` se determina desde `transacciones`:
  - En ventas → `calificacion_comprador_pending`.
  - En compras → `calificacion_vendedor_pending`.
- `resumen` siempre refleja totales globales del usuario, independiente del filtro `tipo` o paginación activa.
- `calificaciones_pendientes` en resumen cuenta todas las transacciones donde el usuario tiene calificación pendiente.
- ENUMs internos nunca salen crudos — `rol` se transforma a `es_tienda_verificada` en `contraparte`.
- Campos excluidos: `telefono`, `correo`, `estado`, `rol` de la contraparte.
- Índices recomendados: `transacciones(vendedor_id, created_at)`, `transacciones(comprador_id, created_at)`.
