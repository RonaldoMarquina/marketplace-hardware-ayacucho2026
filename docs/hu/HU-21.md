# HU-21: Recuperación de Contraseña

**Como** usuario registrado que olvidó su contraseña
**Quiero** recibir un enlace seguro para restablecerla
**Para** recuperar acceso a mi cuenta sin intervención del administrador.

## Endpoints

| Método | Ruta                                    | Acción                        |
|--------|-----------------------------------------|-------------------------------|
| POST   | `/api/v1/auth/password/forgot`          | Solicitar enlace de reseteo   |
| POST   | `/api/v1/auth/password/reset`           | Confirmar nueva contraseña    |

🌐 Ambos públicos — sin JWT

---

## Flujo — Solicitar reseteo (POST)

1. Validar `correo` presente y formato válido (sino 400/422).
2. Buscar usuario por correo en BD.
3. Si no existe → responder 200 igualmente (no revelar si el correo está registrado).
4. Si existe → verificar estado:
   - `ACTIVO` → continuar.
   - `PENDIENTE_VERIFICACION` → 200 sin enviar correo — el usuario debe verificar su cuenta primero.
   - `EN_REVISION` → 200 sin enviar correo — cuenta pendiente de aprobación admin.
   - `BLOQUEADO` → 200 sin enviar correo — no revelar que está bloqueado.
   - `RECHAZADO` → 200 sin enviar correo.
5. Invalidar tokens anteriores del mismo tipo: `UPDATE SET usado=True WHERE usuario_id AND tipo="PASSWORD_RESET" AND usado=False`.
6. Generar token con `secrets.token_hex(32)`.
7. INSERT en `tokens_verificacion`:
   - `usuario_id, token, tipo="PASSWORD_RESET"`
   - `expira_en=NOW()+1h, usado=False`.
8. Enviar correo async con enlace `https://[dominio_frontend]/reset-password?token=...`.
9. Si falla envío → loggear, NO revelar el error al usuario.
10. Rate limit: máx 3 solicitudes por IP/correo cada 15 min (sino 429).

### Body

```json
{ "correo": "juan@gmail.com" }
```

### Respuesta 200 OK (siempre, salvo rate limit)

```json
{
  "mensaje": "Si el correo está registrado y la cuenta está activa, recibirás un enlace en los próximos minutos."
}
```

### 429 Too Many Requests

```json
{
  "error": "RATE_LIMIT_PASSWORD_RESET",
  "mensaje": "Demasiadas solicitudes. Espera 15 minutos antes de intentar de nuevo.",
  "disponible_en": "2026-06-20T15:00:00"
}
```

## Flujo — Confirmar nueva contraseña (POST)

1. Validar `token` y `password` presentes (sino 400).
2. Buscar token en `tokens_verificacion` con `tipo="PASSWORD_RESET"` (sino 404).
3. Verificar `usado=False` (sino 409).
4. Verificar `expira_en > NOW()` (sino 410).
5. Verificar que el usuario sigue `ACTIVO` (sino 403 — estado cambió desde que se solicitó el reset).
6. Validar complejidad de `password`: mínimo 8 chars, al menos 1 mayúscula, 1 número, 1 carácter especial (sino 422).
7. Verificar que `password` nuevo es diferente al hash actual (sino 409).
8. Ejecutar en **transacción SQLAlchemy**:
   - `UPDATE usuarios SET password=bcrypt.hash(password, salt=10)`.
   - `UPDATE tokens_verificacion SET usado=True`.
   - `UPDATE tokens_verificacion SET usado=True WHERE usuario_id AND tipo="PASSWORD_RESET" AND usado=False` — invalidar cualquier otro token activo.
   - Resetear `intentos_fallidos=0` y `bloqueado_hasta=NULL` — consistente con HU-04.
9. Log: `usuario_id, ip, timestamp, accion="PASSWORD_RESET"`.

### Body

```json
{
  "token": "a3f2c1...",
  "password": "NuevaPass123!"
}
```

### Respuesta 200 OK

```json
{
  "mensaje": "Contraseña actualizada correctamente. Ya puedes iniciar sesión.",
  "usuario_id": 7
}
```

---

## Errores

| Código | Situación |
|--------|-----------|
| 400 | `correo` ausente o inválido (forgot) / `token` o `password` ausentes (reset) |
| 403 | Usuario no `ACTIVO` al momento de confirmar el reset |
| 404 | Token no existe |
| 409 | Token ya usado / nueva contraseña igual a la actual |
| 410 | Token expirado |
| 422 | Formato correo inválido / password no cumple requisitos de complejidad |
| 429 | Más de 3 solicitudes en 15 minutos |

---

## Reglas

- **Seguridad ante todo**: el endpoint `/forgot` siempre responde 200 — nunca revela si un correo existe, si la cuenta está bloqueada ni ningún otro estado interno.
- Token generado con `secrets.token_hex(32)` → 64 chars hex — consistente con HU-03.
- Token de un solo uso — `usado=True` tras reset exitoso.
- Vigencia del token: **1 hora** — más corta que HU-03 (24h) porque implica cambio de credencial.
- Solo un token activo por usuario por tipo — nueva solicitud invalida tokens anteriores.
- `tipo="PASSWORD_RESET"` reutiliza tabla `tokens_verificacion` de HU-03 — sin tabla extra.
- Complejidad de password consistente con HU-01 y HU-02: mínimo 8 chars, mayúscula, número, carácter especial.
- Nueva contraseña debe ser diferente a la actual — verificado contra el hash en BD.
- Reset exitoso resetea `intentos_fallidos=0` y `bloqueado_hasta=NULL` — consistente con HU-04, el usuario queda desbloqueado temporalmente.
- Envío de correo siempre async con try/except independiente — fallo no bloquea ni revela error.
- Rate limit consultado desde `tokens_verificacion` — sin Redis en MVP, consistente con HU-03 y HU-12.
- `BLOQUEADO_TEMP` (HU-04) → el reset exitoso lo libera vía reseteo de `intentos_fallidos`.
- `BLOQUEADO` por admin (HU-20) → el reset no lo libera, el estado lo controla solo el admin.
- Log de reset en `admin_log` no aplica — es acción del usuario, va en log general de la aplicación.
- `/uploads/` y `.env` en `.gitignore`.
- Password nunca se expone en ninguna respuesta.
