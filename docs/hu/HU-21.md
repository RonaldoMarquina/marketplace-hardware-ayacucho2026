# HU-21: Recuperacion de Contrasena

## Objetivo

- Actor: usuario registrado que olvido su contrasena
- Capacidad: solicitar y completar un reseteo seguro de credencial
- Resultado esperado: recuperar acceso a la cuenta sin intervencion manual del
  administrador

## Interfaz

| Metodo | Ruta | Finalidad |
|--------|------|-----------|
| `POST` | `/api/v1/auth/password/forgot` | Solicitar enlace de reseteo |
| `POST` | `/api/v1/auth/password/reset` | Confirmar nueva contrasena |

Ambos endpoints son publicos.

## Operacion 1: solicitar reseteo

### Entrada

```json
{ "correo": "usuario@correo.com" }
```

### Flujo principal

1. validar presencia y formato de `correo`
2. buscar usuario por correo
3. responder de manera uniforme si el correo no existe o la cuenta no puede
   continuar con el flujo
4. permitir emision de token solo para cuentas aptas
5. invalidar tokens anteriores del mismo tipo
6. generar nuevo token de reseteo
7. registrar token con vigencia corta
8. enviar correo con enlace de reseteo
9. aplicar limite de solicitudes por ventana de tiempo

### Respuesta esperada

`200 OK` con mensaje generico de envio potencial del enlace.

## Operacion 2: confirmar nueva contrasena

### Entrada

```json
{
  "token": "a3f2c1...",
  "password": "NuevaPass123!"
}
```

### Flujo principal

1. validar presencia de `token` y `password`
2. verificar existencia del token con `tipo = PASSWORD_RESET`
3. verificar que el token no haya sido usado
4. verificar que el token no haya expirado
5. verificar que el usuario asociado siga en estado compatible
6. validar complejidad de la nueva contrasena
7. impedir reutilizar la misma contrasena actual
8. actualizar hash de contrasena
9. invalidar tokens de reseteo activos del usuario
10. resetear contadores de bloqueo temporal ligados al login
11. registrar el evento de reseteo

### Respuesta esperada

`200 OK` con confirmacion de cambio exitoso.

## Datos relevantes

### Tabla implicada

- `tokens_verificacion`

### Tipo de token

- `PASSWORD_RESET`

## Errores

| Codigo | Situacion |
|--------|-----------|
| `400` | campos ausentes |
| `403` | cuenta no habilitada para confirmar reseteo |
| `404` | token inexistente |
| `409` | token usado o nueva contrasena igual a la actual |
| `410` | token expirado |
| `422` | formato de correo o contrasena invalida |
| `429` | limite de solicitudes de reseteo alcanzado |

## Reglas

- el endpoint de solicitud no debe revelar si el correo existe
- el token de reseteo debe ser aleatorio, de un solo uso y mas corto en
  vigencia que el de verificacion de correo
- un reseteo exitoso invalida otros tokens activos del mismo tipo
- un reseteo exitoso puede limpiar bloqueos temporales asociados a intentos de
  login
- un bloqueo administrativo no debe levantarse mediante este flujo
- la contrasena nunca debe exponerse ni directa ni indirectamente en respuestas
