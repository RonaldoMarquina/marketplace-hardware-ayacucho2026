# HU-13: Panel de Moderación

**Como** administrador
**Quiero** revisar y gestionar anuncios reportados
**Para** proteger a los compradores y mantener la confianza.

## Endpoints

| Método | Ruta                                          | Acceso   |
|--------|-----------------------------------------------|----------|
| POST   | `/api/v1/anuncios/:anuncio_id/reportar`       | 🔒 JWT   |
| GET    | `/api/v1/admin/anuncios/reportados`           | 🔒 ADMIN |
| PATCH  | `/api/v1/admin/anuncios/:anuncio_id/bloquear`    | 🔒 ADMIN |
| PATCH  | `/api/v1/admin/anuncios/:anuncio_id/desbloquear` | 🔒 ADMIN |

Middlewares encadenados en rutas `/admin/*`: `jwt_required()` → `check_admin_role()`

---

## Flujo — Reportar (POST)

1. Validar `anuncio_id` entero positivo (sino 400).
2. Verificar JWT válido + cuenta `ACTIVO` (sino 401/403).
3. Verificar anuncio existe y `estado=ACTIVO` (sino 404).
4. Verificar comprador ≠ propietario del anuncio (sino 409).
5. Verificar que el usuario no tiene reporte `PENDIENTE` ni `REVISADO` sobre este mismo anuncio en su estado actual — es decir, si el anuncio fue desbloqueado después del reporte anterior, se permite reportar de nuevo (sino 409).
6. Consultar `reportes` del usuario hoy → si ≥ 10 → 429.
7. Validar `motivo` en ENUM con Marshmallow (sino 422).
8. INSERT en `reportes` con `estado=PENDIENTE`.
9. **No cambiar estado del anuncio** — decisión exclusiva del admin.

## Flujo — Lista reportados (GET)

1. Verificar JWT + rol ADMIN (sino 401/403).
2. Validar `page` y `limit` (sino 400).
3. Query: anuncios con al menos 1 reporte `PENDIENTE`, agrupados por `anuncio_id`.
4. LEFT JOIN usuarios → obtener `vendedor_nombre` y `vendedor_id`.
5. Ordenar: `total_reportes DESC`, luego `ultimo_reporte DESC`.
6. Paginar con OFFSET.

## Flujo — Bloquear (PATCH)

Ejecutar en **transacción SQLAlchemy**:
1. Verificar JWT + rol ADMIN (sino 401/403).
2. Verificar anuncio existe (sino 404).
3. Verificar `estado=ACTIVO` (sino 409).
4. Validar `motivo_admin` no vacío, max 500 chars (sino 400).
5. `UPDATE anuncios SET estado="BLOQUEADO"`.
6. `UPDATE reportes SET estado="REVISADO" WHERE anuncio_id AND estado="PENDIENTE"`.
7. `INSERT INTO moderacion_log (accion="BLOQUEADO", admin_id, anuncio_id, motivo_admin, created_at)`.
8. Si falla cualquier paso → ROLLBACK completo.

## Flujo — Desbloquear (PATCH)

Ejecutar en **transacción SQLAlchemy**:
1. Verificar JWT + rol ADMIN (sino 401/403).
2. Verificar anuncio existe (sino 404).
3. Verificar `estado=BLOQUEADO` (sino 409).
4. Validar `motivo_admin` no vacío, max 500 chars (sino 400).
5. `UPDATE anuncios SET estado="ACTIVO"`.
6. `INSERT INTO moderacion_log (accion="DESBLOQUEADO", admin_id, anuncio_id, motivo_admin, created_at)`.
7. Reportes anteriores permanecen `estado="REVISADO"` — no eliminar.
8. Si falla cualquier paso → ROLLBACK completo.

## ENUMs

**motivo (usuario):** FRAUDE | PRECIO_ENGAÑOSO | PRODUCTO_FALSO | CONTENIDO_INAPROPIADO | DUPLICADO | OTRO

---

## Errores

| Código | Situación |
|--------|-----------|
| 400 | `anuncio_id` inválido / `motivo_admin` vacío o > 500 chars / `page` inválido |
| 401 | Sin JWT o token inválido |
| 403 | Rol insuficiente / cuenta no ACTIVO |
| 404 | Anuncio inexistente o no ACTIVO (reportar) / inexistente (bloquear-desbloquear) |
| 409 | Autoreporte / reporte duplicado sobre anuncio activo / anuncio ya en estado objetivo |
| 422 | `motivo` fuera de ENUM |
| 429 | ≥ 10 reportes realizados hoy por el usuario |

---

## Esquema `moderacion_log`

id             → PK
admin_id       → FK usuarios
anuncio_id     → FK anuncios
accion         → ENUM: BLOQUEADO | DESBLOQUEADO
motivo_admin   → TEXT (obligatorio en ambas acciones)
created_at     → TIMESTAMP

## Esquema `reportes`

id             → PK
comprador_id   → FK usuarios
anuncio_id     → FK anuncios
motivo         → ENUM
estado         → ENUM: PENDIENTE | REVISADO
created_at     → TIMESTAMP

---

## Reglas

- Anuncio `BLOQUEADO` invisible en feed, búsquedas, detalle y contacto — para todos incluyendo al propietario.
- `estado_anuncio` se expone en endpoints `/admin/*` como excepción explícita a la regla general — los admins necesitan el valor real.
- `motivo_admin` obligatorio tanto al bloquear como al desbloquear — toda acción admin queda justificada en `moderacion_log`.
- Reporte duplicado: se bloquea si el usuario ya tiene reporte `PENDIENTE` o `REVISADO` sobre el anuncio en su ciclo activo actual. Si el anuncio fue desbloqueado después, se permite reportar de nuevo.
- Límite de 10 reportes por usuario por día calendario — consultado desde tabla `reportes`.
- `moderacion_log` es auditoría permanente, nunca se elimina ni se modifica.
- Reportes anteriores permanecen `REVISADO` tras desbloqueo — no se eliminan ni se revierten a `PENDIENTE`.
- Sin notificación al vendedor en este MVP.
- Historial admin: `GET /api/v1/admin/moderacion/historial`.
