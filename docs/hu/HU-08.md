# HU-08: Gestion de Estado / Borrado Logico de Anuncio

**Como** vendedor autenticado  
**Quiero** marcar anuncios como vendido, desactivarlos o reactivarlos  
**Para** controlar su disponibilidad sin eliminarlos fisicamente.

## Nota de integracion

Parte de esta HU ya habia sido adelantada en la **HU-07**, donde se implementaron:

- `PATCH /api/v1/anuncios/:anuncio_id/desactivar`
- `PATCH /api/v1/anuncios/:anuncio_id/reactivar`

La **HU-08** consolida la gestion de estados del anuncio y agrega formalmente:

- `PATCH /api/v1/anuncios/:anuncio_id/vendido`

Por compatibilidad con la implementacion previa, la reactivacion conserva tambien la regla de la HU-07:

- maximo **3 reactivaciones por anuncio** durante toda su vida util.

## Endpoints

PATCH `/api/v1/anuncios/:anuncio_id/vendido`  
PATCH `/api/v1/anuncios/:anuncio_id/desactivar`  
PATCH `/api/v1/anuncios/:anuncio_id/reactivar`  
JWT requerido

## Transiciones permitidas

```text
ACTIVO    -> VENDIDO    (/vendido)
ACTIVO    -> INACTIVO   (/desactivar)
INACTIVO  -> ACTIVO     (/reactivar)
VENDIDO   -> [terminal, ninguna]
BLOQUEADO -> [ninguna para el vendedor]
```

## Flujo comun

1. Validar JWT.
2. Verificar anuncio existe; si no, responder `404`.
3. Verificar propietario; si no coincide con el usuario autenticado, responder `403`.
4. Verificar que la transicion sea valida segun el estado actual; si no, responder `409` o `403`.
5. Actualizar `estado` y `updated_at`.
6. Registrar log tecnico con `usuario_id`, `anuncio_id`, `estado_anterior`, `estado_nuevo` y timestamp.

## Regla adicional en `/reactivar`

Antes de reactivar:

- si el usuario es `USER_ESTANDAR`, debe tener **menos de 25 anuncios activos**
- si el usuario es `TIENDA_VERIFICADA`, no tiene limite de anuncios activos
- el anuncio no puede haber alcanzado su limite de **3 reactivaciones**

## Endpoint: marcar como vendido

### Request

PATCH `/api/v1/anuncios/:anuncio_id/vendido`

### Flujo especifico

1. Verificar que el anuncio este en estado `ACTIVO`.
2. Cambiar `estado` a `VENDIDO`.
3. Conservar el resto de la informacion del anuncio y su media.

### Respuesta

**200 OK**
```json
{
  "id": 15,
  "estado": "VENDIDO",
  "updated_at": "2026-01-01T00:00:00",
  "message": "Tu anuncio ha sido marcado como vendido exitosamente."
}
```

## Endpoint: desactivar

### Request

PATCH `/api/v1/anuncios/:anuncio_id/desactivar`

### Flujo especifico

1. Verificar que el anuncio este en estado `ACTIVO`.
2. Cambiar `estado` a `INACTIVO`.

### Respuesta

**200 OK**
```json
{
  "id": 15,
  "estado": "INACTIVO",
  "updated_at": "2026-01-01T00:00:00"
}
```

## Endpoint: reactivar

### Request

PATCH `/api/v1/anuncios/:anuncio_id/reactivar`

### Flujo especifico

1. Verificar que el anuncio este en estado `INACTIVO`.
2. Verificar limite de anuncios activos del usuario segun su rol.
3. Verificar que `reactivaciones_count < 3`.
4. Cambiar `estado` a `ACTIVO`.
5. Incrementar `reactivaciones_count`.

### Respuesta

**200 OK**
```json
{
  "id": 15,
  "estado": "ACTIVO",
  "reactivaciones_count": 1,
  "reactivaciones_restantes": 2,
  "updated_at": "2026-01-01T00:00:00"
}
```

## Errores

- `403` No es propietario.
- `403` El anuncio esta `BLOQUEADO`.
- `403` El usuario `USER_ESTANDAR` ya tiene 25 anuncios activos al reactivar.
- `403` El anuncio ya agoto sus 3 reactivaciones.
- `404` Anuncio no existe.
- `409` Transicion invalida, por ejemplo:
  - `VENDIDO -> VENDIDO`
  - `INACTIVO -> VENDIDO`
  - `ACTIVO -> REACTIVAR`

## Reglas

- Nunca ejecutar `DELETE`; siempre se usa borrado logico por `estado`.
- `VENDIDO` e `INACTIVO` no deben aparecer en feed ni en busquedas publicas.
- `VENDIDO` es estado terminal para el vendedor.
- La media y el historial del anuncio se conservan.
- `reactivaciones_count` vive en la tabla `anuncios` y se inicializa en `0`.
- Las reglas de la HU-07 y HU-08 conviven:
  - limite de 3 reactivaciones por anuncio
  - limite de 25 anuncios activos para `USER_ESTANDAR`
  - sin limite de anuncios activos para `TIENDA_VERIFICADA`
