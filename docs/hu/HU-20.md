# HU-20: Gestion de Usuarios desde Administracion

## Objetivo

- Actor: administrador
- Capacidad: listar, revisar, activar, rechazar, bloquear y desbloquear cuentas
- Resultado esperado: mantener integridad operativa y control sobre usuarios y
  tiendas verificadas

## Interfaz

| Metodo | Ruta | Finalidad |
|--------|------|-----------|
| `GET` | `/api/v1/admin/usuarios` | Listar usuarios |
| `GET` | `/api/v1/admin/usuarios/:usuario_id` | Consultar detalle |
| `PATCH` | `/api/v1/admin/usuarios/:usuario_id/activar` | Activar cuenta en revision |
| `PATCH` | `/api/v1/admin/usuarios/:usuario_id/rechazar` | Rechazar tienda en revision |
| `PATCH` | `/api/v1/admin/usuarios/:usuario_id/bloquear` | Bloquear cuenta activa |
| `PATCH` | `/api/v1/admin/usuarios/:usuario_id/desbloquear` | Desbloquear cuenta bloqueada |

Todas las rutas exigen JWT valido y rol `ADMIN`.

## Operacion 1: listar usuarios

### Parametros

| Parametro | Alcance |
|-----------|---------|
| `estado` | filtro por estado |
| `rol` | filtro por rol |
| `q` | busqueda por nombre o correo |
| `page` | paginacion |
| `limit` | tamano de pagina |

### Flujo principal

1. validar JWT y rol `ADMIN`
2. validar parametros de consulta
3. consultar usuarios con filtros opcionales
4. unir datos de tienda cuando aplique
5. ordenar por `created_at DESC`
6. paginar

### Respuesta esperada

`200 OK` con lista de usuarios y bloque de paginacion.

## Operacion 2: consultar detalle de usuario

### Flujo principal

1. validar JWT y rol `ADMIN`
2. validar `usuario_id`
3. verificar existencia del usuario
4. consultar datos base del usuario
5. unir informacion de tienda cuando corresponda
6. consultar reputacion como vendedor y comprador
7. consultar ultimas acciones administrativas asociadas

### Respuesta esperada

`200 OK` con informacion consolidada del usuario.

## Operacion 3: activar cuenta

### Flujo principal

1. validar JWT y rol `ADMIN`
2. validar `usuario_id`
3. verificar existencia del usuario
4. permitir la operacion solo si `estado = EN_REVISION`
5. permitir la operacion solo para `rol = TIENDA_VERIFICADA`
6. actualizar `estado = ACTIVO`
7. registrar accion `USUARIO_ACTIVADO` en `admin_log`

### Respuesta esperada

`200 OK` con `usuario_id`, `estado`, `accion`, `admin_id` y `updated_at`.

## Operacion 4: rechazar tienda

### Entrada

```json
{ "motivo": "Documento de identidad ilegible." }
```

### Flujo principal

1. validar JWT y rol `ADMIN`
2. validar `usuario_id`
3. verificar existencia del usuario
4. permitir la operacion solo si `estado = EN_REVISION`
5. permitir la operacion solo para `rol = TIENDA_VERIFICADA`
6. validar `motivo`
7. actualizar `estado = RECHAZADO`
8. registrar accion `TIENDA_RECHAZADA` en `admin_log`

### Respuesta esperada

`200 OK` con `usuario_id`, `estado`, `motivo`, `accion`, `admin_id` y
`updated_at`.

## Operacion 5: bloquear cuenta

### Entrada

```json
{ "motivo": "Multiples reportes confirmados." }
```

### Flujo principal

1. validar JWT y rol `ADMIN`
2. validar `usuario_id`
3. verificar existencia del usuario
4. permitir la operacion solo si `estado = ACTIVO`
5. impedir bloqueo de cuentas con rol `ADMIN`
6. validar `motivo`
7. actualizar `estado = BLOQUEADO`
8. desactivar anuncios `ACTIVO` del usuario dentro de la misma operacion
9. registrar accion `USUARIO_BLOQUEADO` en `admin_log`

### Respuesta esperada

`200 OK` con `usuario_id`, `estado`, `anuncios_desactivados`, `motivo`,
`accion`, `admin_id` y `updated_at`.

## Operacion 6: desbloquear cuenta

### Entrada

```json
{ "motivo": "Reportes no confirmados." }
```

### Flujo principal

1. validar JWT y rol `ADMIN`
2. validar `usuario_id`
3. verificar existencia del usuario
4. permitir la operacion solo si `estado = BLOQUEADO`
5. validar `motivo`
6. actualizar `estado = ACTIVO`
7. registrar accion `USUARIO_DESBLOQUEADO` en `admin_log`
8. no reactivar automaticamente los anuncios desactivados por el bloqueo

### Respuesta esperada

`200 OK` con `usuario_id`, `estado`, `motivo`, `accion`, `admin_id` y
`updated_at`.

## Datos relevantes

### Tabla de auditoria

- `admin_log`

### Acciones esperadas en `admin_log`

- `USUARIO_ACTIVADO`
- `TIENDA_RECHAZADA`
- `USUARIO_BLOQUEADO`
- `USUARIO_DESBLOQUEADO`

## Errores

| Codigo | Situacion |
|--------|-----------|
| `400` | identificador, motivo o parametros invalidos |
| `401` | sin JWT o token invalido |
| `403` | rol insuficiente |
| `404` | usuario inexistente |
| `409` | estado actual incompatible con la accion |
| `422` | accion no aplicable al rol o intento de bloquear administrador |

## Reglas

- los endpoints administrativos exponen `rol` y `estado` reales por necesidad
  operativa
- bloquear un usuario desactiva sus anuncios `ACTIVO`, pero no los elimina
- desbloquear un usuario no reactiva anuncios automaticamente
- `motivo` es obligatorio para rechazar, bloquear y desbloquear
- `admin_log` debe conservarse como auditoria permanente
- una tienda `RECHAZADO` no se reactiva por este flujo; requiere un nuevo
  proceso de registro o revision segun la politica del proyecto
