# HU-20: Gestión de Usuarios desde Admin

**Como** administrador
**Quiero** gestionar cuentas de usuarios y verificar tiendas
**Para** mantener la integridad de la plataforma y activar tiendas verificadas.

## Endpoints

| Método | Ruta                                              | Acción                        |
|--------|---------------------------------------------------|-------------------------------|
| GET    | `/api/v1/admin/usuarios`                          | Listar usuarios               |
| GET    | `/api/v1/admin/usuarios/:usuario_id`              | Ver detalle de usuario        |
| PATCH  | `/api/v1/admin/usuarios/:usuario_id/activar`      | Activar cuenta EN_REVISION    |
| PATCH  | `/api/v1/admin/usuarios/:usuario_id/rechazar`     | Rechazar tienda EN_REVISION   |
| PATCH  | `/api/v1/admin/usuarios/:usuario_id/bloquear`     | Bloquear cuenta ACTIVO        |
| PATCH  | `/api/v1/admin/usuarios/:usuario_id/desbloquear`  | Desbloquear cuenta BLOQUEADO  |

Middlewares encadenados en todas las rutas: `jwt_required()` → `check_admin_role()`

---

## Flujo — Listar usuarios (GET)

1. Verificar JWT + rol ADMIN (sino 401/403).
2. Validar query params.
3. Query `usuarios` con filtros opcionales.
4. LEFT JOIN `tiendas` para usuarios `TIENDA_VERIFICADA`.
5. Paginar `ORDER BY created_at DESC`.

### Query params

| Param       | Valores                                                              | Default  |
|-------------|----------------------------------------------------------------------|----------|
| estado      | `ACTIVO` \| `PENDIENTE_VERIFICACION` \| `EN_REVISION` \| `BLOQUEADO` | todos    |
| rol         | `USER_ESTANDAR` \| `TIENDA_VERIFICADA`                               | todos    |
| q           | Busca en nombre y correo, 2–100 chars                                | —        |
| page        | Entero > 0                                                           | 1        |
| limit       | Entero 1–50                                                          | 20       |

## Flujo — Activar cuenta (PATCH)

Ejecutar en **transacción SQLAlchemy**:
1. Verificar JWT + rol ADMIN (sino 401/403).
2. Validar `usuario_id` entero positivo (sino 400).
3. Verificar usuario existe (sino 404).
4. Verificar `estado=EN_REVISION` (sino 409).
5. Verificar `rol=TIENDA_VERIFICADA` — solo tiendas pasan por `EN_REVISION` (sino 422).
6. `UPDATE usuarios SET estado="ACTIVO"`.
7. `INSERT INTO admin_log (accion="USUARIO_ACTIVADO", admin_id, usuario_id, created_at)`.
8. Si falla cualquier paso → ROLLBACK.

### Respuesta 200 OK

```json
{
  "usuario_id": 2,
  "nombre_comercial": "TechStore Ayacucho",
  "estado": "ACTIVO",
  "accion": "USUARIO_ACTIVADO",
  "admin_id": 1,
  "updated_at": "2026-06-20T10:00:00"
}
```

## Flujo — Rechazar tienda (PATCH)

Ejecutar en **transacción SQLAlchemy**:
1. Verificar JWT + rol ADMIN (sino 401/403).
2. Validar `usuario_id` entero positivo (sino 400).
3. Verificar usuario existe (sino 404).
4. Verificar `estado=EN_REVISION` (sino 409).
5. Verificar `rol=TIENDA_VERIFICADA` (sino 422).
6. Validar `motivo` no vacío, max 500 chars (sino 400).
7. `UPDATE usuarios SET estado="RECHAZADO"`.
8. `INSERT INTO admin_log (accion="TIENDA_RECHAZADA", admin_id, usuario_id, motivo, created_at)`.
9. Si falla cualquier paso → ROLLBACK.

## Flujo — Bloquear cuenta (PATCH)

Ejecutar en **transacción SQLAlchemy**:
1. Verificar JWT + rol ADMIN (sino 401/403).
2. Validar `usuario_id` entero positivo (sino 400).
3. Verificar usuario existe (sino 404).
4. Verificar `estado=ACTIVO` (sino 409).
5. Verificar que el usuario no es ADMIN (sino 422 — no se puede bloquear a otro admin).
6. Validar `motivo` no vacío, max 500 chars (sino 400).
7. `UPDATE usuarios SET estado="BLOQUEADO"`.
8. `UPDATE anuncios SET estado="INACTIVO" WHERE usuario_id AND estado="ACTIVO"`.
9. `INSERT INTO admin_log (accion="USUARIO_BLOQUEADO", admin_id, usuario_id, motivo, created_at)`.
10. Si falla cualquier paso → ROLLBACK.

### Body

```json
{ "motivo": "Múltiples reportes de fraude confirmados." }
```

### Respuesta 200 OK

```json
{
  "usuario_id": 1,
  "nombre": "Juan Quispe",
  "estado": "BLOQUEADO",
  "anuncios_desactivados": 3,
  "motivo": "Múltiples reportes de fraude confirmados.",
  "accion": "USUARIO_BLOQUEADO",
  "admin_id": 1,
  "updated_at": "2026-06-20T11:00:00"
}
```

## Flujo — Desbloquear cuenta (PATCH)

Ejecutar en **transacción SQLAlchemy**:
1. Verificar JWT + rol ADMIN (sino 401/403).
2. Validar `usuario_id` entero positivo (sino 400).
3. Verificar usuario existe (sino 404).
4. Verificar `estado=BLOQUEADO` (sino 409).
5. Validar `motivo` no vacío, max 500 chars (sino 400).
6. `UPDATE usuarios SET estado="ACTIVO"`.
7. `INSERT INTO admin_log (accion="USUARIO_DESBLOQUEADO", admin_id, usuario_id, motivo, created_at)`.
8. Si falla cualquier paso → ROLLBACK.
- Anuncios desactivados por el bloqueo NO se reactivan automáticamente — el usuario debe reactivarlos manualmente desde HU-07.

## Errores generales

| Código | Situación |
|--------|-----------|
| 400 | `usuario_id` inválido / `motivo` ausente o > 500 chars / `q` < 2 chars |
| 401 | Sin JWT o token inválido |
| 403 | Rol insuficiente — no es ADMIN |
| 404 | Usuario no existe |
| 409 | Usuario ya en el estado objetivo / estado incorrecto para la acción |
| 422 | Acción no aplica al rol del usuario / intento de bloquear a otro admin |

## Esquema `admin_log`

id          → PK
admin_id    → FK usuarios
usuario_id  → FK usuarios (nullable — para acciones sobre anuncios como HU-13)
anuncio_id  → FK anuncios (nullable — para acciones sobre anuncios)
accion      → ENUM: USUARIO_ACTIVADO | TIENDA_RECHAZADA | USUARIO_BLOQUEADO | USUARIO_DESBLOQUEADO | BLOQUEADO | DESBLOQUEADO
motivo      → TEXT nullable
created_at  → TIMESTAMP

---

## Reglas

- Middlewares encadenados en todas las rutas: `jwt_required()` → `check_admin_role()`.
- `rol` y `estado` se exponen crudos en endpoints `/admin/*` — excepción explícita, el admin necesita los valores reales.
- Admin no puede bloquearse a sí mismo ni a otro admin — verificado en paso 5 del flujo de bloquear.
- Bloquear usuario → sus anuncios `ACTIVO` pasan a `INACTIVO` automáticamente en la misma transacción.
- Desbloquear usuario → anuncios NO se reactivan automáticamente — el usuario decide desde HU-07.
- `motivo` obligatorio en rechazar, bloquear y desbloquear — toda acción admin queda justificada.
- `motivo` no requerido en activar — la activación es una aprobación sin necesidad de justificación.
- `admin_log` unifica el log de HU-13 (`moderacion_log`) y el nuevo log de usuarios — misma tabla, `usuario_id` o `anuncio_id` nullable según la acción.
- `admin_log` es auditoría permanente — nunca se elimina ni se modifica.
- Estado `RECHAZADO` es terminal — una tienda rechazada no puede ser reactivada desde este endpoint; debe registrarse de nuevo desde HU-02.
- `documento_identidad` solo visible en detalle de usuario para admin — nunca en endpoints públicos.
- Índices recomendados: `usuarios(estado)`, `usuarios(rol)`, `admin_log(usuario_id)`, `admin_log(admin_id)`.
