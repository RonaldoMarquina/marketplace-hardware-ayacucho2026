# HU-04: Autenticacion con Login y JWT

## Objetivo

- Actor: usuario verificado
- Capacidad: autenticarse con correo y contrasena
- Resultado esperado: obtener un JWT valido para acceder a rutas protegidas

## Interfaz

| Metodo | Ruta | Finalidad |
|--------|------|-----------|
| `POST` | `/api/v1/auth/login` | Iniciar sesion |

## Entrada

```json
{
  "correo": "usuario@correo.com",
  "password": "Password123!"
}
```

## Operacion: inicio de sesion

### Flujo principal

1. validar presencia de campos requeridos
2. validar formato de correo
3. buscar usuario por correo
4. responder de forma generica si las credenciales no coinciden
5. incrementar intentos fallidos ante password incorrecto
6. bloquear temporalmente la cuenta al alcanzar el umbral de intentos
7. evaluar el estado actual de la cuenta antes de generar token
8. resetear contadores al autenticarse correctamente
9. generar JWT
10. registrar el intento de autenticacion

## Datos del token

### Payload esperado

```json
{
  "sub": 1,
  "correo": "usuario@correo.com",
  "rol": "USER_ESTANDAR",
  "iat": 1234567890,
  "exp": 1234567890
}
```

### Restricciones

- vigencia del token: `8h`
- secreto tomado desde variables de entorno
- sin refresh token en el alcance actual

## Respuestas esperadas

### Login exitoso

`200 OK` con token y datos del usuario autenticado.

### Cuenta pendiente de verificacion

`403` con indicacion de que la cuenta debe verificar correo.

### Cuenta en revision

`403` con indicacion de revision administrativa pendiente.

### Cuenta bloqueada

`403` si el bloqueo es administrativo.

### Bloqueo temporal por intentos

`429` con fecha disponible para nuevo intento.

## Datos relevantes

### Campos auxiliares en `usuarios`

- `intentos_fallidos`
- `bloqueado_hasta`

## Errores

| Codigo | Situacion |
|--------|-----------|
| `400` | campos ausentes o vacios |
| `401` | credenciales invalidas |
| `403` | cuenta no habilitada para login |
| `422` | formato de correo invalido |
| `429` | limite de intentos fallidos alcanzado |

## Reglas

- no se debe revelar si el correo existe o no
- `BLOQUEADO_TEMP` se libera automaticamente al vencer `bloqueado_hasta`
- `BLOQUEADO` requiere intervencion administrativa
- un login exitoso reinicia `intentos_fallidos`
- el endpoint puede exponer `rol`, `estado` y `es_tienda_verificada` porque se
  trata de informacion del propio usuario autenticado
