# HU-04: Autenticación (Login + JWT)

**Como** usuario verificado
**Quiero** iniciar sesión con correo y contraseña
**Para** recibir un JWT y acceder a funciones protegidas.

## Endpoint

POST `/api/v1/auth/login`

## Entrada

- correo (formato email válido)
- password

## Flujo (orden obligatorio)

1. Validar campos presentes y no vacíos con Marshmallow (sino 400).
2. Validar formato correo (sino 422).
3. Buscar usuario por correo en BD.
4. Si no existe → 401 con mensaje genérico (no revelar que el correo no existe).
5. `bcrypt.check_password_hash(hash, password)`.
6. Si no coincide → incrementar `intentos_fallidos` + 1.
   - Si `intentos_fallidos` >= 5 → `UPDATE estado="BLOQUEADO_TEMP"`, registrar `bloqueado_hasta=NOW()+15min` → 429.
   - Si < 5 → 401 con mensaje genérico.
7. Verificar estado del usuario:
   - `ACTIVO` → continuar.
   - `PENDIENTE_VERIFICACION` → 403.
   - `EN_REVISION` → 403.
   - `BLOQUEADO` → 403.
   - `BLOQUEADO_TEMP` → verificar `bloqueado_hasta`:
     - Si `bloqueado_hasta > NOW()` → 429 con `disponible_en`.
     - Si `bloqueado_hasta <= NOW()` → resetear `intentos_fallidos=0`, `estado=ACTIVO`, continuar.
8. Resetear `intentos_fallidos=0` tras login exitoso.
9. Generar JWT con Flask-JWT-Extended.
10. Log: `usuario_id, ip, timestamp, exitoso=True/False`.

## JWT payload

```json
{
  "sub": 1,
  "correo": "juan@gmail.com",
  "rol": "USER_ESTANDAR",
  "iat": 1234567890,
  "exp": 1234567890
}
```

- `exp = NOW() + 8h`
- `JWT_SECRET` en `.env`, nunca hardcodeado.

## Respuesta

**200 OK**
```json
{
  "token": "eyJ...",
  "nombre": "Juan Quispe",
  "correo": "juan@gmail.com",
  "rol": "USER_ESTANDAR",
  "es_tienda_verificada": false,
  "estado": "ACTIVO",
  "expira_en": "8h"
}
```

**403 — cuenta no verificada**
```json
{
  "error": "CUENTA_NO_VERIFICADA",
  "mensaje": "Debes verificar tu correo antes de iniciar sesión.",
  "puede_reenviar": true
}
```

**403 — cuenta en revisión**
```json
{
  "error": "CUENTA_EN_REVISION",
  "mensaje": "Tu solicitud de tienda está siendo revisada por un administrador.",
  "puede_reenviar": false
}
```

**403 — cuenta bloqueada**
```json
{
  "error": "CUENTA_BLOQUEADA",
  "mensaje": "Tu cuenta ha sido suspendida. Contacta al administrador.",
  "puede_reenviar": false
}
```

**429 — rate limit por intentos**
```json
{
  "error": "RATE_LIMIT_LOGIN",
  "mensaje": "Demasiados intentos fallidos. Cuenta bloqueada temporalmente.",
  "disponible_en": "2026-06-20T15:00:00"
}
```

## Errores

| Código | Situación |
|--------|-----------|
| 400 | Campos vacíos o ausentes |
| 401 | Credenciales inválidas (mismo mensaje para correo inexistente y password incorrecto) |
| 422 | Formato de correo inválido |
| 403 | Cuenta `PENDIENTE_VERIFICACION` / `EN_REVISION` / `BLOQUEADO` |
| 429 | 5 o más intentos fallidos consecutivos — bloqueo temporal 15 min |

## Esquema campos en `usuarios` para rate limiting

intentos_fallidos  → TINYINT default 0
bloqueado_hasta    → TIMESTAMP nullable

## Reglas

- Mensaje idéntico para correo inexistente y password incorrecto — prevenir enumeración de usuarios.
- Password nunca se expone en ninguna respuesta.
- Rate limit implementado en BD — sin Redis en MVP, consistente con HU-12 y HU-13.
- 5 intentos fallidos consecutivos → `BLOQUEADO_TEMP` por 15 minutos.
- `BLOQUEADO_TEMP` se libera automáticamente al cumplirse `bloqueado_hasta` — sin intervención admin.
- `BLOQUEADO` (por admin desde HU-13) es permanente — no se libera automáticamente.
- `intentos_fallidos` se resetea a 0 tras login exitoso.
- Sin refresh token en este MVP — el usuario debe hacer login nuevamente al expirar.
- JWT expira en 8h — `JWT_SECRET` en `.env`, nunca hardcodeado.
- `rol` se expone crudo en este endpoint y en el JWT payload — es el propio usuario viendo su información, no un endpoint público.
- `es_tienda_verificada` también se expone en la respuesta — para que el frontend pueda configurar la UI inmediatamente sin consulta adicional.
- `estado` se expone en la respuesta — es el propio usuario viendo su información, no aplica la regla de transformación de endpoints públicos.
- Middleware `jwt_required()` en todos los endpoints protegidos.
- Log de intentos: `usuario_id, ip, timestamp, exitoso` — para auditoría futura.
