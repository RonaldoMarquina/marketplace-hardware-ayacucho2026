# HU-03: Verificación de Correo Electrónico

**Como** usuario recién registrado
**Quiero** recibir un correo con enlace seguro
**Para** activar mi cuenta y acceder a la plataforma.

## Endpoints

GET `/api/v1/auth/verify-email?token=XXXXXX`
POST `/api/v1/auth/verify-email/resend`

---

## Flujo — Envío automático (post-registro)

Se dispara automáticamente al completar HU-01 o HU-02 de forma async:

1. Generar token con `secrets.token_hex(32)`.
2. INSERT en `tokens_verificacion`:
   - `usuario_id, token, tipo="EMAIL_VERIFICATION"`
   - `expira_en=NOW()+24h, usado=False`.
3. Enviar correo con enlace `https://[dominio_frontend]/verificar?token=...`.
4. Si falla el envío → loggear error, NO revertir el registro.

## Flujo — Verificación (GET)

1. Validar que `token` está presente en query param (sino 400).
2. Buscar token en `tokens_verificacion` (sino 404).
3. Verificar `usado=False` (sino 409).
4. Verificar `expira_en > NOW()` (sino 410).
5. Verificar estado actual del usuario:
   - `USER_ESTANDAR` con `PENDIENTE_VERIFICACION` → `UPDATE estado="ACTIVO"`.
   - `TIENDA_VERIFICADA` con `EN_REVISION` → `UPDATE estado="EN_REVISION"` — el correo queda verificado pero el estado no cambia, la activación la hace el admin.
6. `UPDATE tokens_verificacion SET usado=True`.
7. Retornar 200 con mensaje diferenciado según rol.

## Flujo — Reenvío (POST)

1. Validar `correo` presente en body (sino 400).
2. Buscar usuario por correo (sino 404).
3. Verificar estado == `PENDIENTE_VERIFICACION` o `EN_REVISION` (sino 409).
4. Invalidar tokens anteriores del mismo tipo: `UPDATE SET usado=True WHERE usuario_id AND tipo="EMAIL_VERIFICATION" AND usado=False`.
5. Generar nuevo token con `secrets.token_hex(32)`.
6. INSERT en `tokens_verificacion` con `expira_en=NOW()+24h`.
7. Enviar correo async.
8. Rate limit: máx 3 reenvíos por IP/usuario cada 15 min (sino 429).

---

## Respuestas

**200 OK — verificación exitosa USER_ESTANDAR**
```json
{
  "mensaje": "Correo verificado. Tu cuenta está activa, ya puedes iniciar sesión.",
  "estado": "ACTIVO"
}
```

**200 OK — verificación exitosa TIENDA_VERIFICADA**
```json
{
  "mensaje": "Correo verificado. Tu solicitud de tienda está siendo revisada por un administrador.",
  "estado": "EN_REVISION"
}
```

**200 OK — reenvío exitoso**
```json
{
  "mensaje": "Correo de verificación reenviado. Revisa tu bandeja de entrada.",
  "expira_en": "2026-06-21T14:30:00"
}
```

**429 Too Many Requests — rate limit reenvío**
```json
{
  "error": "RATE_LIMIT_REENVIO",
  "mensaje": "Demasiados intentos. Espera 15 minutos antes de solicitar otro correo.",
  "disponible_en": "2026-06-20T15:00:00"
}
```

---

## Errores

| Código | Situación |
|--------|-----------|
| 400 | `token` ausente en GET / `correo` ausente en POST |
| 404 | Token no existe / usuario no encontrado en reenvío |
| 409 | Token ya usado / cuenta ya activa (reenvío a cuenta ACTIVO) |
| 410 | Token expirado |
| 429 | Más de 3 reenvíos en 15 minutos |

---

## Esquema `tokens_verificacion`

id          → PK
usuario_id  → FK usuarios
token       → UNIQUE, 64 chars hex
tipo        → ENUM: EMAIL_VERIFICATION
expira_en   → TIMESTAMP
usado       → BOOLEAN default FALSE
created_at  → TIMESTAMP

---

## Reglas

- Token generado con `secrets.token_hex(32)` → 64 chars hex — nunca predecible.
- Token de un solo uso — `usado=True` tras verificación exitosa o reenvío.
- Solo un token activo por usuario por tipo — reenvío invalida todos los anteriores.
- Vigencia 24h desde generación.
- Índice en columna `token` en BD — se consulta en cada verificación.
- Envío de correo siempre async con try/except independiente — fallo no bloquea ni revierte nada.
- **USER_ESTANDAR**: verificar correo → `estado=ACTIVO` → puede hacer login y publicar.
- **TIENDA_VERIFICADA**: verificar correo → `estado` permanece `EN_REVISION` → no puede hacer login hasta que admin apruebe.
- Cuenta ya `ACTIVO` solicitando reenvío → 409, no enviar correo.
- Cuenta `BLOQUEADA` solicitando reenvío → 409, no enviar correo.
- `estado` se expone en las respuestas de este endpoint porque es información propia del usuario — no aplica la regla de transformación de endpoints públicos.
- Rate limit de reenvío consultado desde tabla `tokens_verificacion` — sin Redis en MVP.
