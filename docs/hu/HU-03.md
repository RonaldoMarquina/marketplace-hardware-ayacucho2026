# HU-03: Verificacion de Correo Electronico

## Objetivo

- Actor: usuario recien registrado
- Capacidad: recibir y usar un enlace seguro de verificacion
- Resultado esperado: confirmar el correo y habilitar el flujo de activacion de
  la cuenta

## Interfaz

| Metodo | Ruta | Finalidad |
|--------|------|-----------|
| `GET` | `/api/v1/auth/verify-email?token=...` | Verificar correo |
| `POST` | `/api/v1/auth/verify-email/resend` | Reenviar correo de verificacion |

## Operacion 1: emision automatica post-registro

Esta operacion se dispara automaticamente despues de HU-01 o HU-02.

### Flujo principal

1. generar token aleatorio de verificacion
2. registrar token en `tokens_verificacion` con `tipo = EMAIL_VERIFICATION`
3. asignar vigencia de `24h`
4. enviar correo con enlace de verificacion
5. conservar el registro de usuario aunque falle el envio de correo

## Operacion 2: verificar correo

### Flujo principal

1. validar presencia de `token`
2. verificar existencia del token
3. verificar que el token no haya sido usado
4. verificar que el token no haya expirado
5. identificar estado y rol del usuario asociado
6. si el usuario es `USER_ESTANDAR` con cuenta pendiente, actualizar a
   `ACTIVO`
7. si el usuario es `TIENDA_VERIFICADA`, conservar `EN_REVISION` y registrar
   solo la verificacion de correo
8. marcar el token como usado

### Respuestas esperadas

- `200 OK` para usuario estandar con cuenta activada
- `200 OK` para tienda con correo verificado y revision administrativa pendiente

## Operacion 3: reenviar verificacion

### Entrada

```json
{ "correo": "usuario@correo.com" }
```

### Flujo principal

1. validar presencia de `correo`
2. buscar usuario por correo
3. permitir la operacion solo para estados compatibles con verificacion pendiente
4. invalidar tokens anteriores del mismo tipo
5. generar un nuevo token
6. registrar nuevo token con vigencia de `24h`
7. enviar correo de forma asincrona
8. aplicar limite de reenvios por ventana de tiempo

### Respuesta esperada

`200 OK` con mensaje de reenvio y fecha de expiracion del nuevo token.

## Datos relevantes

### Tabla implicada

- `tokens_verificacion`

### Tipo de token

- `EMAIL_VERIFICATION`

## Errores

| Codigo | Situacion |
|--------|-----------|
| `400` | token o correo ausente |
| `404` | token inexistente o usuario no encontrado para reenvio |
| `409` | token usado o cuenta no apta para reenvio |
| `410` | token expirado |
| `429` | limite de reenvios alcanzado |

## Reglas

- el token debe ser aleatorio, de un solo uso y no predecible
- solo puede existir un token activo por usuario y tipo
- reenviar invalida tokens anteriores del mismo tipo
- la verificacion de correo no activa automaticamente una tienda en revision
- el envio de correo no debe revertir el registro del usuario si falla
- el control de reenvio se mantiene en la base de datos dentro del alcance del
  MVP
