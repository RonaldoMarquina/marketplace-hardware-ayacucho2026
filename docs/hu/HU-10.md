# HU-10: Filtrado Técnico de Anuncios

**Como** comprador con requerimientos técnicos
**Quiero** filtrar por categoría, subcategoría, condición, precio y specs
**Para** encontrar el producto compatible sin revisar uno por uno.

## Endpoint

GET `/api/v1/anuncios/buscar`
🌐 Público — sin JWT

## Query params

| Param           | Validación                                              |
|-----------------|---------------------------------------------------------|
| categoria       | ENUM válido                                             |
| subcategoria    | Alfanumérico + espacios, max 50 chars, case-insensitive |
| condicion       | ENUM válido                                             |
| precio_min      | DECIMAL > 0, independiente de precio_max                |
| precio_max      | DECIMAL > 0, independiente de precio_min                |
| specs           | Objeto clave-valor, ver formato abajo                   |
| q               | 2–100 chars, escapar `% _ \`                           |
| order_by        | `precio_asc`, `precio_desc`, `reciente`, default=`reciente` |
| page            | Entero > 0, default=1                                   |
| limit           | Entero 1–50, default=20                                 |

### Formato de specs (múltiples simultáneas)

GET /api/v1/anuncios/buscar?subcategoria=Procesador&specs[socket]=AM4&specs[nucleos]=6&specs[incluye_cooler]=true

- Cada `specs[clave]` filtra dentro del JSON de `especificaciones`.
- Todas las specs se encadenan con AND.
- Claves: alfanumérico + guion bajo, max 50 chars.
- Valores: string, número o booleano. Se compara como string contra `JSON_EXTRACT`.
- Máximo 10 specs simultáneas por petición.

## Flujo

1. Validar `page`, `limit`, `order_by`.
2. Validar `categoria` y `condicion` contra ENUM (sino 422).
3. Validar `subcategoria` — solo alfanumérico + espacios (sino 422).
4. Validar `precio_min` y `precio_max` individualmente > 0.
5. Si ambos presentes, validar `precio_min` ≤ `precio_max` (sino 400).
6. Validar cada clave de `specs[]` — solo alfanumérico + guion bajo (sino 422).
7. Validar que no supera 10 specs simultáneas (sino 422).
8. Construir query base: `WHERE estado='ACTIVO'`.
9. Encadenar filtros con AND según params presentes:
   - `AND categoria = :categoria`
   - `AND subcategoria = :subcategoria COLLATE utf8mb4_unicode_ci`
   - `AND condicion = :condicion`
   - `AND precio >= :precio_min` (si solo viene precio_min)
   - `AND precio <= :precio_max` (si solo viene precio_max)
   - `AND precio BETWEEN :precio_min AND :precio_max` (si vienen ambos)
   - Por cada `specs[clave]=valor`:
     `AND JSON_UNQUOTE(JSON_EXTRACT(especificaciones, '$.clave')) = :valor`
   - `AND (titulo LIKE :q OR descripcion LIKE :q OR subcategoria LIKE :q) COLLATE utf8mb4_unicode_ci`
10. Aplicar ORDER BY según `order_by`:
    - `reciente` → `ORDER BY created_at DESC`
    - `precio_asc` → `ORDER BY precio ASC`
    - `precio_desc` → `ORDER BY precio DESC`
11. OFFSET = (page - 1) * limit.
12. INNER JOIN usuarios → transformar rol a `es_tienda_verificada`.
13. Subquery imagen_principal (`es_principal=True`).
14. Usar **prepared statements** para todos los valores, nunca interpolación.
15. `filtros_aplicados` lista solo params recibidos sin nulls, incluyendo specs expandidas.

## Errores

- 400 `precio_min` > `precio_max`.
- 400 `q` menor a 2 chars.
- 422 `categoria` o `condicion` fuera de ENUM.
- 422 `subcategoria` con caracteres no permitidos.
- 422 Clave de `specs[]` con caracteres no permitidos.
- 422 Más de 10 specs simultáneas.
- 422 `order_by` fuera de valores permitidos.
- 422 Intento de inyección SQL en cualquier param.

## Reglas

- Sin resultados → 200 con `data: []`, nunca 404.
- Sin filtros → comportamiento idéntico a HU-09 con `order_by=reciente`.
- Todos los filtros son opcionales y se pueden combinar libremente.
- `precio_min` y `precio_max` son independientes — pueden venir solos o juntos.
- `subcategoria` case-insensitive: "procesador", "Procesador" y "PROCESADOR" resuelven igual.
- `subcategoria` y `categoria` independientes: se pueden usar juntos o por separado.
- `q` busca en `titulo`, `descripcion` y `subcategoria` simultáneamente.
- `q` sanitizado: escapar `%`, `_`, `\` antes del LIKE.
- `specs[clave]` inexistente en JSON → `JSON_EXTRACT` retorna NULL → resultado vacío, no es error.
- ENUMs internos nunca salen crudos — `rol` se transforma a `es_tienda_verificada`.
- `updated_at` puede ser null si el anuncio nunca fue editado → frontend debe manejarlo.
- Campos excluidos: `especificaciones`, `usuario_id`, `telefono`, `estado`, `rol`.
- Prepared statements obligatorios para todos los valores sin excepción.
- Índice compuesto `(estado, created_at)` en tabla `anuncios`.
- Índice en columna `subcategoria`.
- Índice funcional por spec frecuente: `JSON_EXTRACT(especificaciones, '$.socket')`, `JSON_EXTRACT(especificaciones, '$.vram_gb')` — evaluar según volumen de búsquedas.
